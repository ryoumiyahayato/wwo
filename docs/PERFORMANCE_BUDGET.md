# 性能预算

## 目标规模

- 控制单元：64 至 96。
- 行政地区：8。
- 背景人物：100 至 200，不创建可见节点。
- 活跃人物：最多 20，每人最多一个主要行动。
- 关系：仅按需创建。

## 更新预算

- 社会模拟不在 `_process()` 或 `_physics_process()` 执行全量遍历。
- UI 只在数据变化或有限刷新频率下更新。
- 每小时不得深度扫描全世界；邻接预存，前线增量更新。
- 正常游戏不得频繁出现超过 100 ms 的主线程停顿。
- 8 倍速不得持续冻结 UI。
- 目标规模一年 Headless 模拟应低于 10 秒；每日、月度和单次保存结算应低于 100 ms。

## 存档与日志预算

- 历史版本 1 样本：121 个稳定人物、80 个控制单元、8 个组织、1 条关系、1 条进行中行动和 1 条退出历史，带缩进 JSON 约 234 KiB。
- 新增背景成长核心、NPC 进行中行动、生命周期与资金审计后，必须重新测量当前存档体积。
- 日志支持等级过滤，正式运行不输出逐小时海量日志。

## 已测环境

- Windows 11 x86-64。
- Godot `4.6.3.stable.official.7d41c59c4`，Compatibility。

## 2026-07-13 Codex 影子验证

使用精确 Godot 版本，在提交 `9e92e77c422be782fd21baaa69ee7b41099ce8be` 的临时影子副本中只修复一个强类型调用错误后：

- 目标规模一年模拟约 `1,916 ms`。
- 该结果仍低于 10 秒预算，但约为历史 `846 ms` 的 2.3 倍。
- 影子验证没有修改正式仓库，且随后默认分支又加入技能成长、保证成功、继承约束、NPC 边界结算和更严格存档校验，因此 `1,916 ms` 也不是当前最终基准。

## 当前必须重测的指标

完整运行 `tools/run_validation.ps1` 后，再记录：

- 干净导入、每组测试与启动检查的墙钟时间和退出码。
- 日结算、月结算、自动保存和恢复的最大/平均耗时。
- 30 日和一年模拟总耗时。
- NPC 长期行动数量、关系数量和组织资源范围。
- 存档大小及保存、解析和恢复耗时。
- 1280×720 Compatibility 下的持续帧率与最长主线程停顿。
- Windows Release 导出大小、导出耗时和独立启动时间。

## 已知瓶颈

存档按显式操作或每周边界执行，不在渲染帧或每小时运行。JSON 可读性优先于压缩率。当前新增的 1000 种种子玩法可达性回归和更严格存档验证会增加测试耗时，但不进入正式游戏帧循环。未经当前默认分支复测，不再宣称一年模拟、存档体积或导出性能已经通过。

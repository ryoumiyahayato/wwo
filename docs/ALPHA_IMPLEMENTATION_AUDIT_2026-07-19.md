# 2026-07-19 Alpha 实现审查

## 审查范围

本审查针对提交 `693eee8`、`a05d618`、`9efe247`、`bbdf2b7` 和 `beca5cd` 所形成的架空网格 Alpha。

本次审查以仓库中的生产代码、测试和文档为证据，不把开发方完成报告本身当作证明。当前没有在本审查环境中运行 Windows Godot，因此运行性能数字不作重新确认。

## 结论

完成报告中的下列说法不成立：

```text
完整Alpha功能与性能验收通过。
```

更准确的状态是：

> 仓库新增了一批覆盖面较大的经济、合同、债务、企业、人物、组织政治、AI和存档服务候选，并用架空网格预设与直接服务调用场景进行测试；但正式世界地图被错误替换，普通玩家UI没有覆盖报告声称的完整生命周期，窗口测试也没有验证主要交互。因此该实现不能作为完整可玩Alpha接受。

## 一、正式世界地图发生倒退

### 证据

- `project.godot` 曾把普通入口改为 `scenes/alpha/alpha_menu.tscn`。
- `scripts/alpha/alpha_map_canvas.gd` 明确按10×8计算格子尺寸，并读取 `grid_x`、`grid_y` 绘制80个正交矩形。
- `scripts/alpha/alpha_world_service.gd` 从 `CoreDataSet` 的两国八区控制单元生成正式 `cells`。
- `data/alpha/world.json` 把洛岚联邦、维斯塔联盟、八个架空地区和四个模板城市声明为Alpha世界。
- 原 `docs/DECISIONS.md` 明确将旧10×8网格升格为正式世界，并把 `data/world_map/*` 降为历史回归数据。

### 判定

这是设计权威错误，不是单纯视觉简化。当前修正分支已经：

- 恢复V2.3正式世界地图普通入口；
- 撤销旧网格的正式地位；
- 将架空Alpha入口标记为技术夹具；
- 保留经济和政治代码作为待迁移候选。

## 二、普通UI没有完整企业生命周期

### 证据

`scripts/alpha/alpha_ui_binding.gd` 的普通企业动作只有：

```text
enterprise.inspect
enterprise.buy
enterprise.create
```

普通UI没有对应入口用于：

- 接受或选择订单；
- 采购投入；
- 生产或提供服务；
- 运输和交付；
- 雇佣和外包；
- 注资和融资选择；
- 定价；
- 扩张和收缩；
- 处理违约；
- 破产、出售或解散。

这些方法部分存在于 `AlphaEnterpriseService`，但主要由 `AlphaScenarioRunner` 或AI直接调用。

### 判定

可以说“企业服务层存在多项生命周期方法”，不能说“玩家可以通过普通UI完成完整企业生命周期”。

## 三、政治与腐败UI是固定快捷调用

### 证据

普通组织动作包括 `organization.policy` 和 `organization.corrupt`，但执行时：

- 政策调用 `_start_first_policy`，不是玩家从适用政策、预算、辖区和阻力中选择；
- 腐败固定使用 `corruption:accept_benefit`；
- 金额固定为100；
- 知情者取活跃人物列表的第一个对象；
- 受益对象固定为聚合市场或传入组织。

### 判定

服务层存在政策和腐败记录，不等于普通玩家获得了完整政治选择、交换、调查和反制玩法。

## 四、迁移与跨地区人生轨迹主要由测试脚本驱动

### 证据

`AlphaScenarioRunner._employment_and_migration()`：

1. 直接调用工作、工资、生活支出和借款服务；
2. 直接调用 `labor.migrate()`；
3. 随后又直接写入：

```gdscript
player.country_id = "country:vesta_union"
player.region_id = "region:vesta_silverfield"
player.current_status["city_id"] = "city:starhold"
```

普通UI只提供“前往地点”，没有完整迁移、住房、离职、求职和地区身份转换流程。

### 判定

该场景证明若按预定顺序直接调用服务，可以构造一条测试轨迹；它不能证明普通玩家可自然完成这条人生路径。测试脚本在迁移服务后直接写人物地理字段，也暴露出服务权威边界不完整。

## 五、十个场景测试的断言深度有限

### 证据

`tests/alpha/alpha_cross_system_scenarios_test.gd` 对每个场景主要检查：

- `success == true`；
- 固定种子相同；
- trace非空；
- integrity成功；
- snapshot_errors为空。

具体事实断言大多封装在 `AlphaScenarioRunner._completed()` 的布尔数组中。测试没有通过普通UI执行，也没有验证玩家能在无预知情况下发现和完成流程。

### 判定

这些场景适合服务集成回归，但不能代替完整可玩性验收。

## 六、窗口烟测没有验证交互

### 证据

`tests/alpha/alpha_window_smoke_test.gd` 只做以下事情：

- 加载 `alpha_main.tscn`；
- 采样8秒帧数；
- 检查窗口是否为1280×720；
- 检查平均FPS是否不低于30。

它没有点击、输入或验证：

- 借款；
- 工作；
- 企业；
- 合同；
- 迁移；
- 组织职位；
- 政策；
- 腐败调查；
- 保存退出重启载入。

### 判定

完成报告中“实际窗口完成主要交互检查”的说法没有对应仓库自动测试证据。FPS烟测只能支持窗口可加载与渲染性能。

## 七、UI测试主要验证对象存在，不验证生命周期可玩

### 证据

`tests/alpha/alpha_ui_and_presets_test.gd` 对九类对象主要检查：

- 列表非空；
- 详情非空。

之后检查开发命令可注入资金、价格、工资和真值。测试没有执行普通UI中的主要动作链。

### 判定

“对象有列表和详情”不能等同于“正式对象具有完整可操作界面”。

## 八、AI确实调用部分正式服务，但主动性较浅

### 证据

`AlphaAiService`：

- 根据就业、现金、债务和少量已知字段生成最多6个候选；
- 通常选择优先级最高项；
- 每周通过人物ID哈希和日期进行确定性轮换，偶尔选择次级候选；
- 调用劳动、经济、企业、政治和发展服务。

### 判定

以下声明有一定证据支持：

- AI不是每帧开放式规划；
- AI调用了部分与玩家共用的服务；
- 决策历史有界。

但它更接近有限规则优先级与周期轮换，不足以证明完整的“倾向—利益—议程—事项—行动”人物主动性系统。

## 九、服务层并非全部虚构

以下代码确实具有实际实现价值，应保留迁移：

- `AlphaLedgerService` 的平衡分录和幂等入口；
- `AlphaAssetService` 的所有权与控制权分离；
- `AlphaContractService` 的合同状态与债务余额连接；
- `AlphaEconomyService` 的放款、还款、重组和市场冲击；
- `AlphaEnterpriseService` 的企业创建、权益、资产、订单和经营方法；
- `AlphaLaborService` 的雇佣和工资方法；
- `AlphaCharacterService` 的能力评估与发展计划；
- `AlphaPoliticsService` 的职位、政策、腐败与调查记录；
- `AlphaSaveService` 和迁移代码；
- 分层人物和有界历史设计。

这些内容应迁移到正式世界地图，而不是因为地图错误全部删除。

## 十、当前真实状态

### 已做到

- 新增了一批可复用的服务层代码；
- 建立了固定种子的服务集成场景；
- 加入了存档和完整性检查代码；
- 对架空网格夹具进行了性能压力测试；
- AI能够调用部分服务并保持候选数量有界。

### 未做到或未证明

- 正式真实世界地图上的完整Alpha；
- 普通UI中的完整企业生命周期；
- 普通UI中的迁移人生路径；
- 可自由选择的政策、政治交易和腐败方法；
- 通过普通交互完成的十条跨系统轨迹；
- 实际窗口中的完整流程验证；
- 正式地图规模下的性能验收；
- 真实地图与架空Alpha存档之间可靠的地理迁移。

## 后续顺序

1. 保持V2.3正式地图为普通入口。
2. 设计正式世界ID适配与迁移，不再使用两国八区80格作为运行权威。
3. 将账本、合同、债务、企业、人物和政治服务逐项连接正式国家、地区、城市、地点和交通。
4. 补齐普通UI的企业经营、迁移、职位和政策生命周期。
5. 删除场景脚本对人物地理字段的直接写入。
6. 用普通入口与正式地图重写跨系统集成测试。
7. 最后重新测量性能和人工体验。

在以上工作完成前，项目状态应为：

```text
架空网格Alpha已隔离；服务实现候选保留；正式世界Alpha尚未完成。
```

# V2.3 动态关系、介绍与约见

## 六个关系维度

关系是按有向人物对保存的稀疏记录，不预建全人物矩阵：

| 维度 | 范围 |
|---|---|
| 熟悉度 `familiarity` | 0–1000 |
| 信任 `trust` | -1000–1000 |
| 亲近/疏远 `affinity` | -1000–1000 |
| 紧张 `tension` | 0–1000 |
| 义务 `obligation` | -1000–1000 |
| 互惠 `reciprocity` | -1000–1000 |

记录还包含关系状态、已知联系渠道、上次互动时间、有界互动历史和已处理幂等键。所有变化按
配置限幅并写入原因、来源活动/消息/约见和时间。

## 变化规则

- 面对面：熟悉度 +8、信任 +4、亲近 +2。
- 私人回复：熟悉度 +3、信任 +1、互惠 +2。
- 正式介绍：新关系熟悉度 40。
- 到场约见：熟悉度 +10、信任 +5、互惠 +3。
- 爽约：信任 -8、紧张 +6、互惠 -5。

普通联系冷却为 24 小时。同一互动幂等键只能结算一次；数值和历史均有上限。

## 正式介绍

请求者必须认识中间人，中间人必须认识目标人物，并且请求者尚不认识目标。流程为：

1. 请求者向中间人发送延迟私人消息；
2. 中间人消息送达但未读时不作决定；
3. 中间人实际阅读后按确定性规则同意或拒绝；
4. 同意时生成返回请求者的正式介绍消息；
5. 请求者阅读后才取得目标身份知识并建立熟悉度 40 的关系。

同一请求、决定和介绍消息都有稳定 ID 与幂等键。中间流程不会让无关人物（例如阿尔贝）
自动认识目标。

## 约见与爽约

邀请通过通信发送。接受前验证双方认识地点；接受后在双方现有 V2.2 日程中预留同一地点和
时段，并生成各自旅行建议。取消只释放尚未开始的未来预留；已开始约见不能静默取消。

在预约小时内，双方实际同地则完成；配置的一小时宽限内按实际迟到处理；未到场则只结算一次
爽约后果。到场、迟到、取消和爽约全部保存状态、时间和幂等键。

## NPC 规则

正式 NPC 每 24 小时进行一次有界规划：为工作安排可负担旅行、读取已送达消息、按规则回复、
处理介绍请求和约见，并按 168 小时窗口维护关系。NPC 只使用自己的已知地点、人物、现金和
日程，不瞬移，不在每帧规划。背景人物存在于关系与通信网络，但不创建可见节点。

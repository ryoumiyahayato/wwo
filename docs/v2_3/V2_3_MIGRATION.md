# V2.2 到 V2.3 存档迁移

## 输入与输出

迁移器只接受通过 `GameSaveService.validate_v2_2_snapshot()` 的
`v2_2_life_loop_1` 快照，输出 `v2_3_space_cognition_1`。迁移先在内存中进行；源文件始终
只读，目标完整校验后才由 V2.3 原子保存服务写入。

## 保留内容

- 权威时间、随机种子与随机状态
- 正式人物与当前选择人物
- 合同、工资期、出勤和工资
- 住户现金、库存和完整账本
- 健康、疲劳、压力及其他 V2.2 状态
- 组织、关系、通知和旧幂等历史
- 当前小时以前的活动，活动 ID 增加稳定 `migrated:` 前缀

V2.2 关系的熟悉度和信任迁入六维关系，新增维度按零初始化并限幅。旧地点别名映射到正式
V2.3 地点；无法识别的位置回落到人物住所，并记录迁移边界。

## 新增内容

迁移建立正式人物位置、已知地点/边/交通方式、初始身份知识、动态关系、空消息/约见队列、
背景人物和 NPC 空间计划。当前小时之后的固定 `commute_to_work` / `commute_home` 占位被
删除，并按正式交通图重新生成可负担通勤；过去活动不修改。

迁移摘要保存源/目标版本、迁移时间、源摘要、保留的过去活动数量、固定通勤替换标记和
`source_file_modified: false`。

## 确定性、幂等与失败

人物、关系、活动和路线全部稳定排序，生成 ID 只依赖源快照与稳定字段。同一源快照重复迁移
的逐字段结果一致。迁移失败不写目标、不修改源文件，也不调用当前模拟的恢复入口，因此当前
会话不变。文件迁移成功后，目标路径原有有效文件按 V2.3 原子保存规则保留为 `.bak`。

本轮只保证正式 `v2_2_life_loop_1` 评审存档迁移，不声称支持任意历史开发快照或手工修改的
非正式 Schema。

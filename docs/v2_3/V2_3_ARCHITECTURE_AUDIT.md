# V2.3 空间、认知、通信与关系架构审计

审计基线：`a61c3ca08717a64dc025082e5d0ad0ecace6d622`
审计日期：2026-07-18
指定运行时：Godot `4.6.3.stable.official.7d41c59c4`，Compatibility

## 1. 继续复用的 V2.2 正式服务

- `SimulationClock` 仍是唯一权威时钟。V2.3 只消费其 `hour_advanced` 信号，不创建第二时钟，也不增加逐分钟循环。
- `V2ScheduleService` 仍是人物每小时唯一主活动来源。旅行、等待、到达、写信、阅读和见面作为正式活动写入这套日程，不建立平行日程。
- `V2EmploymentService` 继续负责合同、出勤、缺勤和工资。V2.3 在调用其 `record_hour` 前增加“实际位于工作地点且不在途中”的权威检查。
- `V2HouseholdService` 与 `V2LedgerService` 继续负责住户现金、生活物资和整数生丁账本。交通费使用统一账本的 `transport` 分类。
- `V2ConditionService` 继续负责健康、疲劳、压力及 0—1000 限幅。交通与等待效果通过 V2.3 配置接入，不在 UI 中结算。
- `V2OrganizationActivityService` 继续负责工会参与结算；V2.3 在结算前验证真实地点与实际参加小时。
- `V2NotificationService` 继续作为玩家反馈历史。它不是 V2.3 私人消息系统；正式消息另有收发、投递、已读和回复状态，但重要变化仍可投影成有界通知。
- `V2ReviewSaveService` 的临时文件校验、主存档替换和 `.bak` 保留算法继续复用为 V2.3 原子保存基础。
- `V2LifeLoopInterfaceFinal` 的四角布局、紧凑时间编辑区、Esc 逐层关闭和阻断面板暂停语义继续复用。
- `WorldMapCanvas` 的 Robinson 投影、最大 96 倍缩放、LOD、空间索引、批绘层、可见集缓存和 Dirty Flag 继续复用。

## 2. 需要扩展的服务

- 组合根扩展为 `V2_3LifeLoopSimulation`，在 V2.2 每小时结算边界上依次处理旅行、地点条件、消息到达、阅读/回复、知识时效、约见和事件驱动 NPC 规划。
- 日程服务增加 V2.3 可安排活动类型、成组旅行段、目的活动关联、未来固定通勤清理和预约释放；仍保持一人一小时一个权威主活动。
- 关系服务扩展为熟悉度、信任、亲近/疏远、紧张、义务、互惠、互动历史、状态和可用渠道，并兼容迁入 V2.2 的熟悉度与信任。
- UI 绑定增加人物正式位置、路线预览、消息中心、认知日志、动态联系人、约见和开发者真相视角。
- 世界地图只增加一个 V2.3 本地批绘覆盖层及其本地空间索引，不更动国家几何与行政区缓存。
- 存档服务扩展版本识别、V2.2 内存迁移、V2.3 完整字段校验、备份回退和载入前临时实例验证。

## 3. 被替换的固定通勤占位

V2.2 的 `lille_demo_locations.json/commute_duration_hours` 和
`V2ScheduleService._default_hour()` 生成的固定 `commute_to_work` /
`commute_home` 一小时块只是占位。V2.3 初始化或迁移时：

1. 保留当前小时以前的活动原样；
2. 取消当前小时以后的固定通勤块；
3. 根据合同首个工作小时反向规划最快可负担路线；
4. 为每段写入 `wait_for_transport`、`travel_walk`、
   `travel_urban_transit` 或 `travel_regional_train`；
5. 路段之后写入 `arrive`，返程使用独立 `return_home` 目的计划；
6. 现金不足时回退步行；无法准时到达时保留真实行程并产生迟到或缺勤，
   不修改位置伪装到场。

## 4. 增加地点条件的活动

- `work` / `overtime`：皮埃尔必须在里尔五区工厂，阿尔贝必须在北部省政府办公地点。
- `purchase_food` / `purchase_essentials`：必须在营业中的瓦泽姆市场或声明提供相应服务的地点。
- `union_activity`：必须已知活动、到达金属工人工会会所，并按实际到场小时结算。
- `meet_person`、`social_visit`、`workplace_conversation`、
  `organization_conversation`：双方必须同地、同一小时可用且不在冷却期。
- `write_message`、`read_message`：分别要求可用通信服务/可投递地址和已送达消息。

## 5. 现有关系字段迁移

V2.2 的有向关系记录保持原人物方向和稳定人物 ID：

- `familiarity` 与 `trust` 原值迁入并按新范围限幅；
- 新增 `affinity = 0`、`tension = 0`、`obligation = 0`、
  `reciprocity = 0`；
- `last_contact_datetime` 同时作为初始 `last_interaction_datetime`；
- `recent_interactions` 转为带原因和幂等键的 `interaction_history`；
- 原有联系人默认获得 `local_letter` 与可配置面对面渠道；
- 皮埃尔—让娜使用 V2.3 明确初值覆盖旧 Demo 小数值；阿尔贝不会从世界数据或
  UI 静态名单获得让娜/儒勒关系。

## 6. 有限认知与真实世界状态分离

真实状态只存在于地点、旅行、消息、组织、人物和约见等领域服务中。人物认知只存在于
`KnowledgeService` 的人物索引中，必须带事实、主体、来源、获知时间、可信度、时效、
状态和相关消息。

普通 UI、普通地图和联系人选择只能查询当前观察人物的已知地点、已知边、身份知识和
关系渠道。消息“送达”不写入事实；只有阅读或直接观察才写入知识。开发者真相视角明确
标记并直接查询领域状态，但不修改人物知识。切换评审人物只更换查询主体，不重建或重置
任何权威状态。

## 7. 地图增量显示

V2.3 地点使用当前 `WorldMapCanvas.project_lon_lat()` 投影并锚定 Lille 附近经纬度。
本地覆盖层：

- 仅在高倍率 LOD 查询本地空间索引；
- 一次批绘可见地点、已知边、路线预览、当前路段和人物位置；
- 人物、消息和知识不各自拥有常驻 Node；
- 位置/路线变化只置本地覆盖层 Dirty Flag；
- 消息与知识变化不置国家几何、行政区或交通底图 Dirty Flag；
- 拖动只更新既有 CanvasItem 变换和可见查询，不重投影或三角化。

## 8. V2.2 存档迁移

迁移器先读取并校验 `v2_2_life_loop_1` 快照，在内存深拷贝后构造
`v2_3_space_cognition_1`：

- 保留时间、随机状态、人物、合同、住户、现金、库存、账本、状态、出勤、工资、
  通知和旧幂等历史；
- 初始化正式位置、已知地点/边/人物、消息、知识、约见和 NPC 计划；
- 只替换当前小时之后的固定通勤；
- 使用稳定排序和稳定 ID，使同一输入重复迁移逐字段一致；
- 完整校验通过后才原子写入 V2.3 路径；原 V2.2 文件不改写，已有有效主文件保留为
  `.bak`；
- 迁移或载入失败时不调用当前模拟的恢复方法，因此当前运行状态不变。

## 9. 当前最大风险

1. V2.2 结算在活动开始时直接把人物位置改成活动地点，必须由 V2.3 组合根覆盖，避免
   工作、购买和关系活动继续瞬移。
2. 默认日程提前生成 48 小时，旅行插入必须成组、确定性且能原子拒绝冲突，避免目的
   活动存在而行程只写入一半。
3. V2.2 关系目标曾使用短 ID `jeanne`；V2.3 必须统一稳定人物 ID 并在迁移边界兼容，
   不能让 UI 和服务各自维护联系人名单。
4. 保存加载同时覆盖途中路段、按到达时间排序的消息队列和多类幂等键；恢复校验必须在
   临时实例中完成，避免半恢复。
5. 96 倍地图已经有严格性能预算；本地点覆盖层必须完全隔离于国家几何重建。

## 10. 防止 V2.2 回退

- 不删除、不重命名 V2.2 数据、脚本、场景和测试。
- V2.3 使用子类/组合扩展和独立 `data/v2_3` 配置；V2.2 原入口仍可直接加载以执行回归。
- `project.godot` 若切换主入口，只改 `run/main_scene`，不改 Compatibility、窗口、
  输入或缩放设置。
- 统一验证先执行干净编辑器导入，再执行全部 V2.2 与 V2.3 测试并扫描脚本加载错误。
- 继续运行 V2.2 30 日确定性、存档、UI、性能和旧路径清理守卫。
- 地点、票价、延迟、可信度、时效和关系增量全部配置化；UI 只发命令和读取过滤后的
  只读状态。
- 新服务保持有界历史和索引；禁止在 `_process()` / `_physics_process()` 中遍历人物
  或运行社会模拟。

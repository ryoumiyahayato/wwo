# V2.3 交通图与旅行结算

## 图结构

`data/v2_3/lille_travel_graph.json` 定义 12 个正式节点和 15 条双向边，整图连通。每条边保存
稳定 `edge_id`、两端地点、可用方式、各方式耗时与费用、营业时间、疲劳、压力、所需知识和
启用状态。`TravelGraphService` 初始化时一次建立邻接表、地点索引和交通方式索引；寻路不会
重复扫描 JSON。

## 交通方式

| 方式 | 费用 | 典型营业时间 | 结算特点 |
|---|---:|---|---|
| `walk` | 0 | 全天 | 疲劳最高；非必要长距离受 950 疲劳上限约束 |
| `urban_transit` | 每段 5 生丁 | 05:00–23:00 | 城内一小时级路段；现金不足不可选 |
| `regional_train` | 每段 20 生丁 | 06:00–22:00 | 仅配置的车站—区域中心边；最少两小时 |

费用是 V2.3 Demo 平衡值，不代表完整历史票价模型。

## 确定性寻路

`RoutePlannerService` 只使用人物已知地点、边和方式。候选结果包含节点、路段、方式、出发/
到达时间、总时长、总费用、预计疲劳、等待和失败原因。

- `fastest`：最早到达、其次费用、最后稳定路段 ID 序列。
- `cheapest`：最低费用、其次到达时间、最后稳定路段 ID 序列。
- 营业时间不匹配时计算确定性等待；无法等待或无连通路径时返回失败。
- `fastest` 超出现金时继续搜索可负担候选；没有可负担路线时返回明确失败。
- 完全相同权重按稳定 `edge_id` 排序，配置加载顺序不会改变结果。

## 正式旅行与日程

旅行计划写入现有 `V2ScheduleService`，不是第二套日程。计划由 `plan_route`、可选
`wait_for_transport`、每个 `travel_*` 路段、`arrive` 和目的活动组成，使用
`purpose_activity_id` 关联。整组日程先预检冲突，再提交；提交失败不留下半条路线。

每段开始时：

1. 验证人物仍在正确起点、方式仍可用且现金足够；
2. 以稳定幂等键扣除交通费并写入 V2.2 账本 `transport` 分类；
3. 设置 `waiting` 或 `in_transit`，但不改变到达地点。

每段完成时才更新地点、疲劳和压力。旅行中断保留当前位置语义与原因。工作结束后人物仍在
工作地点，返家需要独立路线；系统不执行隐式瞬移。

## 通勤

V2.3 会移除当前小时之后的 V2.2 固定一小时通勤占位，保留过去活动，并按工作开始时间反向
规划最快可负担路线。现金足够时可选市内交通，不足时回退步行；无法准时到达时保留真实路线，
由出勤系统产生迟到、部分缺勤或缺勤。

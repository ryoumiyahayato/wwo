# 正式社会沙盒测试规程

本规程区分三类证据：自动回归、长时间无人运行、人工窗口判断。人工测试不负责证明内部原子性、确定性和存档幂等；这些必须由自动测试完成。人工测试只判断世界是否值得观察和介入。

## 一、测试环境

- Windows 10/11 x64。
- Godot `4.6.3.stable.official.7d41c59c4`。
- Compatibility 渲染器。
- 仓库使用 `master`，测试前不得停留在旧代理分支。
- 正常入口必须是 `res://scenes/v2_3/v2_3_life_loop_menu.tscn`。

建议先执行：

```powershell
git fetch --prune
git checkout master
git pull --ff-only
git branch -r
```

远程分支正常情况下只应保留 `origin/master`。历史 PR 提交仍保留在 Git 历史中，不需要保留历史开发分支。

## 二、完整自动验证

在仓库根目录运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\run_validation.ps1 `
  -GodotPath "D:\Tools\Godot-4.6.3\Godot_v4.6.3-stable_win64.exe" `
  -StepTimeoutSeconds 220
```

通过标准：

1. 每一步退出码为 0。
2. 输出不得包含 `SCRIPT ERROR`、`Parse Error`、脚本加载失败、类型解析失败或断言失败。
3. V2.2、正式 V2.3 和隔离网格夹具测试全部完成。
4. 最后出现完整验证通过信息。

该命令会覆盖：

- 配置和时间；
- 原子账本；
- 日程、就业、住户和状态；
- 地点、路线和旅行；
- 消息、知识、关系和约见；
- NPC空间生活；
- 正式社会沙盒；
- 社会沙盒八项完成路径；
- 存档迁移、保存恢复和确定性；
- 地图集成；
- 30日与一年性能；
- 正常项目无界面启动；
- 旧网格隔离夹具。

## 三、社会沙盒定向自动测试

需要缩短排错范围时，在 PowerShell 中运行：

```powershell
$Godot = "D:\Tools\Godot-4.6.3\Godot_v4.6.3-stable_win64.exe"

& $Godot --editor --headless --path . --quit
& $Godot --headless --path . --script res://tests/v2_3/v2_3_social_sandbox_test.gd
& $Godot --headless --path . --script res://tests/v2_3/v2_3_social_sandbox_completion_test.gd
& $Godot --headless --path . --script res://tests/v2_3/v2_3_save_load_test.gd
& $Godot --headless --path . --script res://tests/v2_3/v2_3_determinism_test.gd
& $Godot --headless --path . --script res://tests/v2_3/v2_3_performance_guard_test.gd
& $Godot --headless --path . --quit-after 5
```

### 正式符合性测试应证明

`v2_3_social_sandbox_test.gd` 应至少证明：

- 使用 `V23ProductSimulationV2`；
- 社会沙盒引用现有日程、地点、知识、关系、组织和账本对象；
- 方法目录不再把争取职位固定为工作组代表路线；
- 当前有效处境均能形成目标，而不是被旧四目标上限截断；
- 稳定处境重新评估后保持目标 ID；
- 玩家和 NPC 任务具有同一正式结构；
- 人物时段和唯一职位声明为冲突对象；
- 故障提交不会留下组织、关系、资金或虚假事件的部分状态；
- 玩家零输入运行30日后产生有序社会事件；
- 30日内同时存在成功和失败；
- 人物认知不超过客观事件；
- 存档恢复后任务、事件、承诺、证据和职位保持；
- 相同初态和种子产生相同结果。

### 八项完成路径测试应证明

`v2_3_social_sandbox_completion_test.gd` 应至少证明：

- 玩家必须明确选择人物对象；
- 正式绑定公开目标、方法、对象、时间和准备程度；
- 预览检查实际地点与双方路线；
- 确认后建立双方日程和正式旅行；
- 行动到期后形成事件；
- 高准备不产生必胜；
- 没有既有承诺时不能用“偿还人情”反向制造新承诺；
- 新字段可保存。

## 四、GitHub Actions与安装包测试

每次PR和`master`推送都会运行：

1. Godot安装和版本检查；
2. 导出模板下载；
3. 脚本导入；
4. 正式控制、社会沙盒、保存和性能测试；
5. 无界面启动；
6. Windows EXE导出；
7. 原始EXE构建产物上传；
8. 独立作业下载EXE并编译Inno Setup安装包；
9. 安装包构建产物上传；
10. `master`成功时覆盖`prototype-latest`预发布。

发布验收时下载：

- `1900-Prototype-Windows-Installer`；
- 必要时下载 `1900-Windows-Export` 对比直接运行；
- 安装器失败时下载 `1900-Inno-Compiler-Log`。

安装包检查：

1. 在未安装机器上安装。
2. 从开始菜单启动。
3. 确认窗口标题包含当前构建短码。
4. 退出后再次启动。
5. 覆盖安装同一安装标识的新版本。
6. 确认旧安装被覆盖，而不是产生第二套程序目录。
7. 卸载并确认主程序文件被删除。

## 五、人工窗口测试

启动命令：

```powershell
& "D:\Tools\Godot-4.6.3\Godot_v4.6.3-stable_win64.exe" --path .
```

人工测试不是逐按钮清单，也不要求玩家达到指定职位。按照下列观察场景进行。

### 场景A：正常开局和零输入世界

1. 进入正常开局。
2. 不提交任何社会计划。
3. 使用时间加速连续推进至少30个游戏日。
4. 每隔5日查看一次事件、人物、关系、组织职位、承诺和知识变化。

记录：

- 哪些NPC主动形成了目标；
- 哪些行动实际占用时间和地点；
- 是否出现请求、拒绝、帮助、调查、竞争或组织行动；
- 是否同时出现成功和失败；
- 是否发生关系、承诺、工作或职位变化；
- 事件是否能解释“谁、何时、在哪里、做了什么、为何知道”；
- 世界是否在没有玩家推动的情况下仍产生有意义变化。

失败标准：

- 30日只有通勤、睡眠和工资结算；
- 所有NPC重复同一行动；
- 事件只显示抽象数值；
- 世界在玩家不操作时社会状态不变；
- NPC明显使用玩家不知道的信息或全局真相。

### 场景B：玩家制定一个实际社会计划

1. 选择皮埃尔。
2. 在处境面板选择一个当前目标。
3. 选择“主动交谈”“请求帮助”“调查线索”等需要对象的方法。
4. 明确选择让娜、儒勒、路易或其他已知人物。
5. 调整开始时间和准备程度。
6. 阅读预览中的地点、双方路线、开始结束时间、费用、风险和违法标记。
7. 提交计划并推进到任务结束。

通过标准：

- 未选择必要对象时不能提交；
- 预览失败时说明具体地点、路线或日程原因；
- 提交后行动者和对象均出现对应日程；
- 不在当地时出现旅行计划；
- 到期后形成成功、失败、拒绝、受阻或延期事实，而不是静默消失；
- 事件结果进入人物认知，而不是所有人同时全知。

### 场景C：合法行动也可能失败

1. 选择加薪、请求帮助、说服或申请加入组织。
2. 使用不同对象、不同时间和不同准备程度重复尝试。
3. 观察对方关系、组织身份、日程和既有利益是否改变结果。

通过标准：公开合法方法不因“合法”自动成功；失败必须留下可理解事实和后续处境。

### 场景D：非法、秘密或越权尝试

1. 选择行贿、威胁、欺骗、窃取文件、伪造文件、泄露消息或破坏。
2. 检查系统允许尝试，而不是因违法直接禁用。
3. 提交后观察费用、失败、暴露、证据、关系和组织后果。

通过标准：

- 非法行动可以进入正式任务；
- 不自动成功；
- 不能绕过时间、地点、对象和资源；
- 成功与失败都应形成客观事件；
- 人物是否知道该事件取决于目击、消息、调查或记录。

当前限制：法律案件、暴力伤害、完整组织纪律和完整目击范围尚未全部实现，不能把没有出现这些后果判定为测试操作错误。

### 场景E：同一对象与唯一职位冲突

1. 找到一个实际空缺职位。
2. 让两名具有成员身份的人在相近时间争取同一职位。
3. 推进到结算。

通过标准：

- 最终只有一名持有人；
- 失败者得到冲突或前置条件失败事实；
- 不出现重复职位授予；
- 后发生的行动读取前一事件改变后的职位状态。

不要使用调试命令直接授予职位后再声称冲突链通过。

### 场景F：人物认知与客观事实

1. 在开发真相视图查看一项秘密或低可见度事件。
2. 切换到不相关人物的普通视图。
3. 查看该人物的知识和事件列表。
4. 再通过消息、接触或调查让其获得相关信息。

通过标准：

- 普通人物最初看不到未知事件；
- 获知时间与客观发生时间可以不同；
- 知识保留来源、可信度和时间；
- 获知后才影响该人物后续目标。

当前限制：相互矛盾消息、完整错误因果解释和主动欺骗链仍是部分实现。

### 场景G：保存、载入和一次性执行

1. 在一个社会任务到期前保存。
2. 退出程序。
3. 载入存档。
4. 推进到任务结束。
5. 再次保存、载入。

通过标准：

- 任务只执行一次；
- 事件ID不改变；
- 事件不重复追加；
- 费用不重复扣除；
- 关系不重复结算；
- 职位不重复授予；
- 人物知识和消息保持一致。

### 场景H：地图与计划价值

在每个社会计划中检查：

- 地图是否帮助判断人物、地点和路线；
- 是否显示实际时间成本；
- 是否能看出当前计划涉及的地点；
- 是否只显示玩家人物有依据知道的信息；
- 是否能看出准入、风险、线索或可能目击范围。

当前该项只能进行产品判断，自动测试尚未证明地图已达到完整指令要求。

## 六、人工测试最终只回答十个问题

1. 世界是否真正会自行变化？
2. NPC行动是否能够理解？
3. 人物是否像具体的人，而不是脚本节点？
4. 玩家是否能够自由选择方向？
5. 是否仍存在固定任务路线？
6. 正常生活手续是否过多？
7. 事件因果是否清楚？
8. 地图和计划是否有用？
9. 不同方法是否确实产生不同社会后果？
10. 是否愿意继续观察和介入这个世界？

这些问题中任何一项为否，都应作为产品缺陷记录。不要通过提高准备度、寻找唯一正确路线或反复点击来替系统证明成立。

# 因果记录与玩家反馈

状态：设计冻结 v1。所有核心状态必须回答“为什么变成现在这样”，但玩家第一层只处理少量摘要。

## 1. 因果记录

```text
CausalRecord
  record_id
  occurred_hour
  settlement_period_id?
  subject_type
  subject_id
  state_key
  delta_or_transition
  source_type
  source_id?
  counterparty_ids[]
  scope_ids[]
  visibility_rule
  parent_record_ids[]
  summary_key
  detail_parameters
```

数值变化、状态转换、权限授予、知识获得和资金流都产生记录。`summary_key` 配合参数生成本地化文本，正式存档不依赖临时 UI 字符串。

## 2. 账本与状态历史

各系统基于同一记录构建不同视图：

- 财务账本：付款方、收款方、收入、支出、余额。
- 时间账本：义务、计划、通勤、休息和超负荷。
- 状态账本：疲劳、压力、健康的增减来源。
- 关系历史：互动、信任、熟悉、好感和冲突。
- 权限历史：职位、法律、委托和撤销。
- 政策链：提出、批准、预算、执行、地方落实与效果。

记录不等于全部都弹通知。

## 4. 四层信息密度

| 层 | 内容 | 示例 |
| --- | --- | --- |
| 扫视 | 图标、名称、状态、趋势 | 疲劳偏高 ↑；本月结余 +9 |
| 悬停 | 当前值、主要原因、阈值 | 工作 +8、恢复 -7；高负荷线 70 |
| 点击详情 | 完整已知来源、历史和可行动入口 | 时间账本、合同、休整与请假 |
| QA | 内部 ID、公式、所有权威记录、权重 | 仅 Debug/`--qa` |

正式 Release 只能访问前三层。

## 5. 成功与风险

行动第一层使用：条件不足、存在风险、把握较高、保证成功。悬停显示当前有效值、成功线、保证线和最主要限制。若系统存在随机成功，显示真实概率；确定性系统不得伪造百分比。

详情展示能力、准备、资金、关系、组织、状态和阻力来源。已过去区间的上下文快照可用于解释为什么行动没有因新职位而追溯加速。

## 6. 数值结算、通知、事件与新闻

- 数值结算：普通微小变化，进入账本，不主动显示。
- 通知：当前人物直接相关且需要注意的变化，短时显示。
- 事件：要求玩家在期限内决定，可自动暂停。
- 世界新闻：重大公开变化，通过媒体和知识系统传播。

同类普通消息按周期聚合，例如“本月 6 个组织在 5 个地区扩大影响”，不显示几十条重复通知。

## 7. 通知优先级

```text
NotificationView
  source_record_ids[]
  importance
  aggregation_key?
  expires_after_real_seconds
  target_navigation
  requires_acknowledgement
```

普通通知自动淡出；重要通知停留更久；要求决定的事件进入事件队列而非普通通知。点击只打开对应人物、组织、地点、账本或议程，不直接执行行为。

## 8. 建议系统

人物中心最多显示 2～4 条当前建议，来源于权威可达性查询：最大风险、最易改善状态、最合适人物、最匹配组织或可行恢复路线。

建议必须指向真实对象和当前可执行行为，不使用“应该多社交”之类无法验证的泛化教程。

## 9. AI 可解释性

AI 的目标、承诺和计划记录选择原因、预期收益、成本、风险、进度和放弃条件。正式玩家只在合理信息范围内看到外在解释，例如“正在寻找稳定工作”；QA 可以查看完整评分。

## 10. 历史保留与压缩

玩家、重要人物、制度决定、资金大额流动、关系里程碑和世界新闻保留长期记录。普通微小结算按月聚合；聚合必须保存总额、主要来源和原记录数量。

任何仍被行动结果、债务、政策、知识或事件引用的记录不得删除。压缩只在月度或存档维护边界执行，不在渲染帧中执行。

## 11. UI 性能

UI 查询预聚合摘要，不每帧扫描完整历史。状态变化发出窄信号，使对应角落组件、卡片或账本增量刷新。

## 12. 首版与保留接口

首版必须：统一因果记录、财务/时间/状态/关系账本、四层信息、聚合通知、状态建议和最近历史。

只保留接口：复杂统计图、全文检索、玩家自定义报告、跨多年审计导出和可视化因果图编辑器。

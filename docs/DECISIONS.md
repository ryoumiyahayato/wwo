# 架构决定

本文件记录当前仍有效的决定。被后续决定修正的历史描述不再保留为并列规则，避免实现与文档出现两个权威答案。

## 2026-07-11 · 固定引擎、渲染器与平台证据

- 决定：使用标准版 Godot `4.6.3.stable.official.7d41c59c4` 和 Compatibility 渲染器。
- 原因：项目硬性约束，并面向 Windows 10/11 x86-64 与较低配置设备。
- 后续影响：所有场景、脚本、导出和测试以该精确版本为准，不自动升级。平台预设不等于兼容证据；只有对应系统实际导出并运行后才能标记通过。

## 2026-07-11 · 分层、稳定 ID 与数据验证

- 决定：依赖方向固定为“数据配置 → 核心模型/服务 → 模拟 → 只读状态/事件 → UI”，UI 不承载模拟公式。
- 决定：实体使用稳定、可读、可验证的 ID；生成 ID 在命名空间内递增，模型为无 Node、无 UI 依赖的强类型数据对象。
- 决定：正式数据先验证结构、类型、范围和全局 ID，再实例化并验证跨实体引用；错误集中返回，不提供部分可用世界。
- 原因：避免场景节点、对象地址、松散字典和 UI 便利访问成为持久化或权威状态。

## 2026-07-11 · 确定性随机与权威时间

- 决定：所有业务随机通过 `DeterministicRandomService`，保存种子和 RNG 状态，不使用全局随机函数。
- 决定：权威时间使用自 1900-01-01 起的整数游戏小时；现实秒只由薄 `SimulationRunner` 转换，页面复用同一世界时钟。
- 决定：单个小时的当前结算顺序为：

```text
小时 → 日 → 月 → 到期队列事件 → 周/自动存档
```

- 原因：整数时间、稳定插入序号和明确周期顺序使相同输入可复现。自动存档必须包含该小时的日/月领域结算和到期事件。
- 修正：该顺序取代早期“小时 → 日 → 周 → 月 → 队列”的历史描述。

## 2026-07-12 · 正交地图与增量前线

- 决定：正式架空世界使用 10×8 正交控制网格，共 80 单元、8 行政地区、2 个国家。
- 决定：前线以排序后的相邻单元 ID 对为唯一键；初始化全量计算一次，控制者变化后只更新相关邻接。
- 决定：地图由单个自定义 `Control` 绘制，不为每个单元创建可见节点。
- 决定：控制压力读取铁路、地区社会影响、控制单元本地支持、多方向进攻和包围，并受最小/最大倍率限制。
- 原因：保持拓扑清晰、渲染与权威状态分离，并避免每帧或每小时扫描全部地图。

## 2026-07-12 · 配置驱动人物与公开信息边界

- 决定：姓名、职业、人口职业倍率、技能、倾向、资质和年龄范围由 JSON 配置，生成器只消费经过验证的配置和统一随机服务。
- 决定：全人口模式按国家总人口、人口群体人数和群体职业结构抽样；国家选择与人物生成使用隔离随机流。
- 决定：正式 UI 只读取公开/已知字段，隐藏资质、气质权重和真实倾向只在显式开发者视图中显示。
- 原因：保证同种子复现、世界人口结构一致，并防止便利访问泄露隐藏信息。

## 2026-07-12 · 分层人物、AI 与稀疏关系

- 决定：背景人物不运行 AI、不创建可见节点；活跃人物上限为 20，所有非玩家活跃人物必须恰有一个 AI 状态。
- 决定：背景人物降级时保存技能、资质、气质、倾向和随机核心，再次升级保留同一稳定身份与成长历史。
- 决定：关系只在实际接触时创建，并以排序人物对建立唯一索引，不预建 O(n²) 矩阵。
- 决定：NPC 每日推进或开始有限候选的真实长期行动，每月更新长期目标；没有既有关系时也可建立第一条关系。
- 原因：在 100 至 200 人规模下同时保持连续性、确定性和有界性能。

## 2026-07-12 · 长期行动、成长与保证成功

- 决定：行动保存工作量、上次结算小时、有效值和效率；不按渲染帧更新，也不逐小时扫描全部人物。
- 决定：依赖发生变化时，必须先按旧效率结算已经过去的区间，再从边界验证目标、状态和权限，随后中断或切换新上下文。
- 决定：玩家可选择学习任意已有技能；学习行动提高所选技能，其他行动产生主要技能的实践成长。
- 决定：主技能达到精通、准备与资金达到配置阈值时，公式提供明确、可测试的有效值加成，使常规行动存在可证明的保证成功上界。
- 决定：玩家和 NPC 使用相同的费用、工作量、效率、结果阈值和领域写回规则。
- 原因：角色自身必须存在完整成长路径，且“充分准备后必然成功”不能依赖开局抽到高能力角色或继承换人。

## 2026-07-12 · 通用结算与领域写回分层

- 决定：`ActionService` 只结算通用数值、技能成长和阈值结果；社会组合根负责关系、成员、职位、情报、政策和控制等领域写回。
- 决定：数值成功只有在要求的领域写回成功后才保留；领域失败必须撤销成功奖励并应用失败结果。
- 原因：防止“行动显示成功但职位/关系/控制没有变化”以及 UI 直接修改模拟状态。

## 2026-07-12 · 组织、辖区与资源循环

- 决定：加入组织与入口职位分配必须原子成功；职位通过人物—组织—职位索引提供权限。
- 决定：组织每月获得由基础值、规模和影响力计算的收入并受资源上限约束；地区社会和军事支援消耗资源。
- 决定：政策只作用于人物国家的法理或实际控制地区；非政府组织的政策权限只覆盖本地区；军事控制支援只允许政府或军队组织。
- 原因：避免成员/职位半提交、组织资源单向耗尽，以及职位权限被解释为全球权限。

## 2026-07-12 · 社会影响与军事控制双通道

- 决定：政策和社会活动修改 `RegionData.social_influence`；军事控制只通过 `MapControlService` 修改控制单元。
- 决定：控制单元 `social_support` 与地区社会影响都参与压力计算，但不合并为单一所有权字段。
- 原因：法理、军事控制、地区社会影响和本地支持具有不同语义，必须分别保存和显示。

## 2026-07-12 · 生命周期与继承

- 决定：人物按年度增龄，老年降低健康；NPC 和背景人物可退休或死亡，领导空位随后补充。
- 决定：继承候选只来自真实关系和共同组织；继承保留继承者既有关系与组织身份，只按配置继承部分资源、关系和最多有限数量的新组织身份。
- 决定：退休、死亡、长期监禁和严重失势必须由年龄、健康、拘禁、声望或明确权威状态证明；自愿退出保持为常规玩家选择。
- 决定：继承前保存人物、组织、关系和 AI 快照，先释放旧玩家活跃槽位；任一步失败则恢复全部社会状态。
- 原因：继承者是已有历史的人物，不能作为空白容器，也不能允许年轻健康角色伪造高收益退出原因。

## 2026-07-12 · 版本化存档与安全替换

- 决定：版本 1 使用可读 JSON；只保存权威可变状态，不保存节点、对象地址或重复静态拓扑。
- 决定：加载先在临时社会组合根恢复并验证，成功后才替换全局会话；临时应用存档地图进行行动校验，失败则恢复加载前地图。
- 决定：恢复必须验证活跃人物上限、层级标记、激活种子完整性、组织/关系双向索引、活跃 NPC 的完整 AI 覆盖，以及玩家与 NPC 行动实例 ID 全局唯一。
- 决定：文件先写 `.tmp` 并刷新，原文件改名为 `.bak` 后替换；失败时恢复备份。备份测试按 JSON 解析后的语义比较，不比较写入前内存中的整数/浮点表现差异。
- 原因：损坏、旧迁移或手工修改的存档不得在加载后形成合法格式但非法运行态。

## 2026-07-13 · 干净检出验证必须自举并检查日志

- 决定：验证入口先运行 Headless 编辑器导入，生成 Godot 全局 `class_name` 缓存，再运行独立 `--script` 测试。
- 决定：Godot 进程退出码为 `0` 仍不足以判定成功；日志中出现脚本解析、类型解析或脚本加载错误时，验证脚本必须失败。
- 实现：统一使用 `tools/run_validation.ps1`。
- 原因：干净检出与已有 `.godot` 缓存的机器必须得到一致结果，不能把“退出码 0 但主场景未成功解析”记为通过。

## 2026-07-13 · 性能和发布声明只引用当前提交证据

- 决定：历史 564 项、846 ms 一年模拟和旧 Windows 导出只作为历史参考；任何模拟公式、AI、存档或人物结构变化后必须重跑。
- 决定：影子副本和临时单点修复结果必须明确标记，不得当作正式默认分支通过记录。
- 原因：测试数量、性能和导出结果只有在同一提交、同一精确引擎版本和完整命令下才构成发布证据。

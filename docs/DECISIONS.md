# 架构决定

## 2026-07-11 · 固定引擎与渲染器

- 决定：使用标准版 Godot `4.6.3.stable.official.7d41c59c4` 和 Compatibility 渲染器。
- 原因：项目硬性约束，并面向 Windows 10/11 x86-64 与较低配置设备。
- 替代方案：Forward+、Mobile、其他 Godot 版本或 .NET 版；均不采用。
- 后续影响：所有场景、脚本、导出和测试以该精确版本为准，不自动升级。

## 2026-07-11 · 最小分层骨架

- 决定：M0 只建立 UI、无场景核心服务和测试三个实际边界，不提前创建空服务类。
- 原因：保持 UI 与未来模拟分离，同时避免为未进入的里程碑搭建庞大框架。
- 替代方案：一次创建全部未来管理器或把逻辑写入菜单脚本；均会扩大范围或形成耦合。
- 后续影响：M1 从独立 `SimulationClock` 开始，不能让菜单或渲染帧成为权威时间源。

## 2026-07-11 · 自包含测试和集中字符串

- 决定：使用 `SceneTree` 脚本作为无第三方测试运行器；M0 可见字符串集中在 `ui_strings.gd`。
- 原因：满足离线、无未批准插件和未来本地化要求，且保持初始实现可替换。
- 替代方案：第三方测试插件或把文字散落在场景脚本；不采用。
- 后续影响：测试必须返回进程退出码；后续本地化可替换字符串存储而不改变菜单职责。

## 2026-07-11 · M0 导出预设边界

- 决定：提供 Windows Desktop x86-64 预设，但将实际导出与产物兼容性验证留在 M9。
- 原因：项目结构与命令需要明确，同时本轮不得扩展到平台交付。
- 替代方案：M0 下载导出模板并宣称平台通过；违反用户约束和里程碑边界。
- 后续影响：README 和 AGENTS 标记导出命令需要本地模板，未经验证不得声称可用。

## 2026-07-11 · M1 权威时间与事件顺序

- 决定：权威状态使用自起点累计的整数游戏小时和前推格里高利历；现实秒只在独立 Runner 中累计并批量提交整小时。1900 年按世纪规则不是闰年。
- 原因：整数权威状态可避免帧率和时区差异，且无需依赖平台日期 API 对 1970 年以前时间的行为。
- 替代方案：用渲染帧计数、系统本地时间或负 Unix 时间戳；前两者不确定，后者增加不必要的平台边界。
- 后续影响：存档只需记录整数小时和不足一小时的现实时间余量；M8 再固定序列化格式。

## 2026-07-11 · M1 调度队列确定性

- 决定：每个推进小时按“小时 → 日 → 周 → 月 → 到期队列事件”发信号。周从 1900-01-01 00:00 起每 168 小时结算；相同到期小时按插入序号稳定排序，事件 ID 唯一。
- 原因：明确边界和并列事件顺序，保证相同操作序列产生相同结果。
- 替代方案：依赖字典遍历、系统排序稳定性或每帧检查事件；均会削弱确定性或性能边界。
- 后续影响：后续系统必须通过时钟信号或队列接入，不得自行以渲染帧判断日历边界。

## 2026-07-11 · M2 稳定 ID 与纯数据模型

- 决定：所有实体 ID 使用小写 `命名空间:slug`，生成 ID 使用命名空间内八位递增序号。模型是无 Node、无 UI 依赖的强类型 `RefCounted`，并显式转换为字典。
- 原因：ID 可读、可验证且不依赖内存地址；纯数据模型便于测试、索引和后续存档。
- 替代方案：节点路径、对象实例 ID、随机 UUID 或把字段留在松散字典；前两者不能持久化，UUID 破坏操作序列可复现性，松散字典缺少边界。
- 后续影响：加载器在全数据集范围检查 ID 唯一性；M8 存档必须保存 ID 计数器状态。

## 2026-07-11 · M2 两阶段数据验证

- 决定：加载器先验证顶层结构、记录字段和全局 ID，再实例化模型并检查所有跨实体引用；错误集中返回，不用首个异常中止进程。
- 原因：内容数据错误需要可定位、可批量修复，并且不得让游戏崩溃。
- 替代方案：模型构造时隐式容错或遇到首错直接退出；会隐藏错误或降低内容迭代效率。
- 后续影响：M3 正式数据必须通过同一加载器，不能绕开验证直接访问 JSON 字典。

## 2026-07-11 · M2 统一确定性随机服务

- 决定：所有随机调用通过一个封装 Godot `RandomNumberGenerator` 的服务；服务支持固定种子、重设种子、读取和恢复 RNG 状态、选取和洗牌。
- 原因：相同种子和操作顺序必须产生相同结果，且 M8 需要恢复进行中的随机序列。
- 替代方案：业务代码自行创建 RNG、使用全局随机函数或只保存初始种子；都会让调用顺序或加载后序列难以控制。
- 后续影响：后续人物、AI 和事件系统不得直接创建随机生成器；默认种子由配置提供，玩家种子 UI 留待对应里程碑。

## 2026-07-12 · M3 正交控制网格与正式世界规模

- 决定：首版正式架空世界采用 `10×8` 正交控制网格，共 80 单元；东西各 5 列为一国，每两行组成一个行政地区，共 8 地区。
- 原因：满足 64 至 96 单元目标，四邻接明确，便于可靠测试地图边缘、孤立、包围和前线，同时仍能由程序绘制呈现清晰战略地图。
- 替代方案：手绘不规则多边形、真实地理或六角格；前两者增加素材与拓扑风险，六角格对本 Demo 的控制机制没有必要收益。
- 后续影响：正式数据显式保存网格坐标和邻接，渲染尺寸可变但拓扑不从像素位置推断。

## 2026-07-12 · M3 增量前线边集合

- 决定：前线以两个相邻控制单元 ID 排序组成的唯一边键保存。初始化全量计算一次；控制者变化后只更新该单元的邻接边。
- 原因：避免每小时或每帧扫描全地图，并使前线生成、删除和重复边处理可单元测试。
- 替代方案：手工前线、每帧比较全部邻接或把前线烘焙进场景；分别不能动态变化、浪费性能或与权威数据分叉。
- 后续影响：地图渲染只读取服务的边集合；前线与控制服务之间通过信号增量刷新。

## 2026-07-12 · M3 单节点程序地图

- 决定：地图使用一个自定义 `Control` 绘制单元、争夺纹理、城市、铁路和前线，不为 80 个控制单元创建独立可见节点。
- 原因：符合性能禁区并让缩放、拖动和图层顺序集中可控。
- 替代方案：每格一个场景节点或引入第三方地图插件；前者增加节点和信号负担，后者违反插件约束。
- 后续影响：单元点击通过坐标命中测试，UI 面板只在选择或数据变化时更新。

## 2026-07-12 · M4 配置驱动人物生成

- 决定：名字、职业原型、随机模式权重、技能/倾向基础、资质与年龄范围全部由单一 JSON 配置加载；生成器只组合经过验证的配置、正式世界数据、统一 RNG 和稳定 ID 服务。
- 原因：同种子可复现、平衡值集中，并避免人物规则散落在 UI 或多个脚本。
- 替代方案：UI 脚本直接随机字段、职业硬编码分支或为每次生成新建未封装 RNG；都会破坏分层或确定性。
- 后续影响：M4 生成器无场景依赖；M5/M6 只能消费人物数据，不能重新解释初始生成规则。

## 2026-07-12 · M4 正式与开发信息边界

- 决定：人物模型提供显式公开快照，排除隐藏资质、性情权重和真实倾向；正式面板只消费公开/已知字段，开发面板默认隐藏并由单独切换显示。
- 原因：防止 UI 通过便利访问意外泄露隐藏数据，同时保留测试和调试能力。
- 替代方案：把全部模型字典交给 UI 后依靠控件自律，或不存隐藏数据；前者容易泄密，后者无法支持资质与调查系统。
- 后续影响：后续正式 UI 必须使用公开快照或明确白名单；开发工具不得改变正常数据结构。

## 2026-07-12 · M4 轻量运行会话

- 决定：使用无场景依赖的静态 `GameSessionService` 保存当前玩家人物、国家、种子和 RNG 状态，使设置页、地图和人物面板共享同一身份。
- 原因：M4 需要场景间连续人物，但完整存档和世界会话生命周期属于 M8。
- 替代方案：每个场景重新生成人物、把人物节点跨场景搬移或提前实现完整存档管理器；分别破坏身份、耦合场景或超出范围。
- 后续影响：退出进程仍会丢失会话；M8 需把同一字段映射到存档而不是替换人物 ID。

## 2026-07-12 · M5 事件式行动进度

- 决定：行动实例保存已完成工作量、上次结算小时和当前效率；仅在查询、权威时间变化或依赖项变化时按时间差结算，不订阅每帧更新，也不逐小时遍历人物。
- 原因：符合权威时间和性能约束，同时暂停、恢复及帧分块结果可以用整数小时稳定复现。
- 替代方案：每帧累加进度、每小时扫描全部人物或只保存预计完成日期；分别耦合渲染、扩大负载或无法正确处理依赖变化。
- 后续影响：所有会改变效率的系统必须先结算旧区间，再更新输入并重算；M8 保存行动时必须包含上次结算小时。

## 2026-07-12 · M5 确定性阈值与领域效果边界

- 决定：第一版八类行动均使用普通成功与必然成功两个确定性阈值，正式 UI 只显示配置化六档定性把握。人物和控制区现有字段直接应用；依赖 M6/M7 实体的效果以稳定描述记录。
- 原因：设计明确要求常规长期行动可通过准备达到必然成功，并允许本版不实现暗杀等本质随机行动；同时不能为行动结果提前搭建组织、关系或社会影响系统。
- 替代方案：显示百分比并随机掷骰、提前创建 M6 实体或完全不应用结果；分别违反信息规则、里程碑边界或完成定义。
- 后续影响：未来若加入不可控行动，必须使用统一 RNG 并新增独立判定策略；不能改变现有常规行动的必然成功语义。

## 2026-07-12 · M6 轻量背景人物与身份连续升级

- 决定：背景人物使用不含隐藏资质、完整技能或 AI 状态的独立轻量模型；升级时以记录中的稳定激活种子确定性补全活跃模型，覆盖并保留原 ID、姓名、地区、职业、组织和公开经历。降级恢复轻量记录但不创建新身份。
- 原因：满足 100 至 200 人规模和背景人物不运行完整模拟的性能约束，同时用稳定 ID 保证层级转换不是同名替身。
- 替代方案：让全部人物永久持有完整模型、降级时删除后重新随机人物或为每人创建节点；分别浪费内存/决策负载、破坏身份连续或违反性能禁区。
- 后续影响：M8 存档必须区分背景与活跃记录；M7 继承候选必须引用稳定人物 ID，不依赖当前层级。

## 2026-07-12 · M6 有界周期 AI 与按需关系索引

- 决定：只有活跃 NPC 拥有 AI 状态；每日从配置化有限候选中评分选择，每月更新长期目标。关系只在实际接触时创建，并以排序人物对建立唯一索引。
- 原因：避免每小时全人物搜索和 O(n²) 关系矩阵，并让相同输入与调用顺序产生确定结果。
- 替代方案：背景人物也执行完整 AI、每小时枚举全部行动、预建全部人物对关系或使用无序字典同分选择；都会超出性能预算或削弱确定性。
- 后续影响：重大事件可显式请求活跃 NPC 重评，但不得把候选集扩展为全世界扫描；M7/M8 复用关系 ID 和周期状态。

## 2026-07-12 · M7 社会影响与军事控制双通道

- 决定：地区政策和社会活动只修改 `RegionData.social_influence`；扩大控制只通过 `MapControlService` 修改控制单元。组织支援在两个通道上分别检查职位权限并消耗资源。
- 原因：法理、军事控制和社会影响必须保持不同权威字段，人物行动不能因便利接口合并成单一所有权。
- 替代方案：政策行动直接改控制者、用社会影响自动重画军事前线或建立统一 `owner`；都会破坏既有地图语义。
- 后续影响：M8 分别保存地区社会影响与控制单元状态；未来社会影响转化为军事效果必须是显式规则或行动。

## 2026-07-12 · M7 关系网络候选与部分继承

- 决定：继承候选只来自已存在关系和共同组织成员，并以配置化可解释分数稳定排序。继承在原世界内切换玩家人物，只转移部分财富/声望、符合条件的职位及按不同系数缩减的盟友/敌对关系。
- 原因：继承需要源于人物真实经历，同时不得完整克隆前人物或重建世界；稳定排序保证复现。
- 替代方案：全名册随机候选、复制完整人物、开始新世界或依赖内存地址关系；分别削弱社会连续性、违反设计或破坏确定性。
- 后续影响：已退出人物必须保留稳定记录供历史和 M8 存档；M8 只序列化现有连续状态，不重新执行继承。
## 2026-07-12 · M8 版本化快照与服务内恢复

- 决定：存档版本 1 使用可读 JSON；文件层由单一 `GameSaveService` 管理，各权威服务提供自身持久快照和先验证后应用的恢复方法。地图只保存可变字段并在加载后重建前线。
- 原因：避免保存节点、对象地址和重复静态拓扑，同时让时钟、人物层级、组织索引和稳定 ID 的一致性可分别测试。
- 替代方案：序列化场景树、直接保存全部核心配置、加载时重跑生成/继承逻辑；分别会绑定运行实例、放大存档并产生配置分叉，或破坏确定性连续性。
- 后续影响：新增存档版本必须显式迁移或拒绝；不能静默解释未知版本。

## 2026-07-12 · M8 临时文件安全替换与单自动档

- 决定：只允许 `user://` JSON；先写 `.tmp` 并刷新，已有文件改名为 `.bak` 后替换，失败则恢复备份。手动档与自动档各一个固定路径，自动档只在权威周边界写入。
- 原因：验证和写入失败不得损坏已有进度，固定槽位也避免无界文件增长。
- 替代方案：直接覆盖目标文件、每小时自动保存或创建无限历史档；会在中断时留下半文件或造成不必要的主线程与磁盘压力。
- 后续影响：当前只验证 Windows 11 文件语义；其他平台必须真机验证后才可声明兼容。

## 2026-07-12 · M8 开发命令与正式状态共用结构

- 决定：开发模式使用同一套权威模型和存档格式，只增加显式模式标记；全部调试变更经 `DeveloperCommandService` 门控并写入有界结算日志。性能数据聚合为分类计数、最近/累计/最大微秒。
- 原因：避免调试分支形成另一套不可复现的世界结构，同时保持正式模式默认隐藏且不逐小时刷日志。
- 替代方案：UI 直接写模型、开发专用存档或逐事件详细日志；会绕开验证、制造格式分叉或违反性能禁区。
- 后续影响：新增开发命令仍须经过统一服务，UI 只负责参数输入和结果展示。

## 2026-07-12 · M9 行动计算与领域写回分层

- 决定：行动面板保存人物、组织或控制单元的稳定目标 ID；`ActionService` 只结算通用数值，成功后由社会组合根分派关系、成员、职位、情报和地区影响结果。
- 原因：保持既有确定性时间公式与领域实体边界，同时让最终核心循环能从正式行动入口完成。
- 替代方案：继续依赖社会面板直接按钮，或让行动服务直接持有全部社会服务；前者无法完成玩家闭环，后者会形成高耦合组合根。
- 后续影响：新增领域行动必须实现幂等写回并为失败、存档加载和断裂目标提供明确行为。

## 2026-07-12 · M9 平台声明以实际产物为准

- 决定：提供 Windows x86-64、Linux x86-64 与 macOS Universal 预设，但只有成功导出并在对应系统运行后才标记兼容。缺少模板时保留真实失败记录，不下载或替换用户指定 Godot。
- 原因：平台预设不是运行证据，且用户明确禁止重新下载、安装或升级 Godot。
- 替代方案：仅凭编辑器运行或预设存在宣布平台通过；会违反完成定义和平台真实性要求。
- 后续影响：匹配模板安装后已完成 Windows Release 导出、产物日志检查和 Windows 11 真机验收；Windows 10 仍须在对应系统实机执行后才能标记通过。
